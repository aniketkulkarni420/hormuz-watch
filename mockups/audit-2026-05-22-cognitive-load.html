Hormuz Watch — design + data + typography audit

Brutal. Devil's advocate. No code changes yet. Captured 2026-05-22 against the live dashboard at hormuz-watch-2.pages.dev.

TL;DR — three sentences

  • 1. You have a brilliant data pipeline strapped to an undifferentiated wall of 30+ tiles, all rendered at roughly the same visual weight; the dashboard doesn't tell a story, it tells facts in parallel.
  • 2. The first thing to fix is not the sidebar width — it's the absence of a hierarchy. Headings + section grouping + verdict-as-hero will buy 80% of the cognitive-load relief at 20% of the redesign cost.
  • 3. Then collapse ~30 tiles to ~12, and only then reconsider sidebar/map split. Most likely answer: keep the map but shrink it to a 240-300px header strip on desktop, give the analytics surface 70% of the screen. But not yet — fix hierarchy first.

§1 16 findings, ranked brutally

Each finding is real. Each is fixable. None of them require touching the data pipeline.

  • P0   No H1 / H2 / H3 system. Title is a 12-22px sticky in the header; everything below is at the same visual rank. User cannot skim.
  • P0   ~30 tiles all rendered with identical chrome (rblock, 16-18px padding, mono 10px label). Verdict tile and "CAPE reroute static reference" look equally important.
  • P0   The verdict is buried in the right panel as one card among many. The primary user task ("is something happening?") needs the answer as a HERO, not as item #4 in a list.
  • P0   Signal bar has 5 tiles but ranks none of them. BDTI gets the same weight as IRR spread, which gets the same weight as Brent. Trader can't tell what to act on first.
  • P1   No section headers in the right panel. 20+ cards scroll as a flat list with zero grouping. "Markets" + "Geopolitics" + "Macro" + "Conditions" + "India" are mentally separable but visually identical.
  • P1   Map takes ~60% of desktop viewport for a dashboard whose primary task is "verdict + driver explanation." Map is reference, not the answer.
  • P1   Cross-signal verdict (xv-list) is the only multi-factor synthesis tile, but it's visually identical to single-metric tiles. The most analytically dense element is invisible.
  • P1   "Conditions" card combines weather + aircraft + seismic. Three unrelated signals, one tile, no clear interpretation. Reader can't tell what they're collectively saying.
  • P1   News tile shows raw headlines. No sentiment, no clustering, no "the system thinks these 3 matter." Just a list.
  • P1   Brent and WTI are separate tiles. They tell one story ("crude oil + spread"). Should be one composite tile, not two separate ones competing for attention.
  • P1   BDTI has just shipped Brent's daily-history chart but BDTI itself still shows only "value + WoW%". Inconsistent — both deserve the same trend-chart treatment.
  • P1   Currency tile shows IRR spread % but no trajectory. "62% spread" is meaningless without "trending tighter or wider over 4 weeks?"
  • P2   LNG, fertilizer, CAPE-reroute are static reference data. They belong on a "context" sidebar or methodology page, not the live dashboard.
  • P2   Cargo ticker concept is brilliant but visually treated as decoration. It says "this is happening live RIGHT NOW" — should be the dashboard's heartbeat, not a third-class header element.
  • P2   No global "as-of" timestamp. Each tile shows its own freshness (some hide it). Reader doesn't know whether the WHOLE dashboard is current.
  • P2   Tabular vs sans split is muddled — mono labels, mono badges, mono headings, mono numbers. The mono is supposed to align numbers; using it everywhere makes the whole UI read as a developer terminal.

§7 Layout — should we increase rpanel and shrink map?

Your question. Devil's-advocate answer: not yet. Three options:

Option A — keep current widths, fix hierarchy firstRECOMMENDED FIRST

Map ~60% / rpanel 360px. Don't change anything spatially. Apply §3-6 first (hierarchy, hero, consolidate, group). Re-measure cognitive load. If still bad, then resize.

Why: if the problem is "too many tiles," giving them more space doesn't fix the noise — it just gives the noise more room.

Option B — rpanel 360 → 480, map shrinks to ~50%SAFE MIDDLE

More room for the redesigned cards. Brent + Currency + BDTI trend charts get full width. Hero verdict spans more comfortably.

Tradeoff: map becomes secondary visually. Still usable but no longer dominant.

Option C — map collapses to top header strip (240-300px tall), rpanel takes 70%+ANALYTICS-FIRST

Map becomes a slim geographic reference strip above the analytics. Vessels still visible as dots, but the "answer the verdict question" becomes the dominant surface.

Tradeoff: users who came for the map are now scrolling for it. Risk of breaking the geographic-anchor identity.

My recommendation

Ship Option A first (no layout change), then run a 5-user test of the redesign. If users still complain about cognitive load AFTER hierarchy + consolidation, ship Option B. Consider Option C only if research shows the map is not a primary navigational anchor (likely — most users don't pan the map; they just want to see "is anything moving").

§8 Color & semantic audit

Two simultaneous color-coding systems compete:

SystemMaps toWhere used
Green / Amber / RedVerdict severityVerdict tile · cross-signal · health bar
Green / RedPrice delta directionBrent change · BDTI change · WTI change
Green ↑ in a price tile means "price went up" (which could be bullish or bearish depending on context). Green ↑ in the verdict tile means "all clear, NORMAL." Two opposite meanings, same color. Trader has to read each tile twice to figure out what green means here.
Fix: reserve red/amber/green for VERDICT severity only. Use a different palette for price deltas (e.g., blue ↑ / orange ↓ — directional but not semantic). Or drop color from price deltas entirely; rely on ▲ / ▼ arrows + bold numbers.

§9 Mobile — additional brutal findings

Mobile audit is structurally clean (we shipped 6 passes recently), but the cognitive-load problem amplifies:

  • P1   Bottom-tab nav fragments the data across 3 surfaces (Map / Ships / Intel). User loses context switching tabs.
  • P1   Default landing is Map tab. For "what's happening?" task, Intel should be default. Map is exploration.
  • P1   Intel tab = a single long scroll of 20+ cards. Same hierarchy problem amplified on a tiny screen.
  • P2   Signal-bar swipe is now discoverable (fade-mask shipped) but users still see only 2 tiles at a time. Stack vertically on mobile?
Fix: default to Intel tab on mobile. Apply the same §3 section grouping to the Intel scroll (sticky H2 sub-nav at the top). Consider replacing horizontal swipe-bar with a vertical "key signals" header (Brent · BDTI · Verdict in 3 stacked rows above the rest).

§10 If you only do 3 things, do these

  1. Verdict-as-hero (§4) — biggest immediate cognitive-load relief. Pulls the 5-second answer out of the rpanel and puts it above the fold with a 1-sentence "what changed" summary. ROI: 10x.
  2. H2 section grouping in rpanel (§3) — turn the 20-card flat scroll into 4 grouped sections (Markets · Strait activity · Geopolitics · Conditions). User can scan section headers in 3 seconds, drill into a section in another 5. ROI: 5x.
  3. Card consolidation per §5 — merge Brent+WTI, fold weather/seismic/aircraft into a single "operating environment" boolean, move LNG/fertilizer/CAPE to /methodology. ROI: 3x.
