Showcase readiness: NOT YET — 3 visible bugs a sharp viewer will catch in 10 seconds

  • 1. The headline "Transits / 24h" tile shows "--" and "FEED PENDING" while 141 vessels of scrape data exist. Your primary chokepoint metric looks broken on load.
  • 2. The "Dark vessels" tile shows "200" — that's the GFW API result-cap, not 200 dark ships near Hormuz. Same class as the old dark_vessels=947 bug. A maritime-literate viewer will know instantly that 200 dark vessels in Hormuz is absurd.
  • 3. The GFW feed is half-broken — loitering dataset 404s, encounters POST times out (524). The "200" is a partial/capped response, not a real reading.

The underlying data pipeline is healthy (oil, BDTI, currency, verdict all fresh + accurate). These are display bugs on three tiles — fixable in an afternoon. But do not showcase until they're fixed; they undermine credibility on exactly the metrics that make this product unique (vessel + dark-fleet tracking).

§1 Data bugs — ranked

P1 · SHIP-BLOCKERTransits tile shows "--" / "FEED PENDING" despite live scrape data

The signal-bar #sigVessels tile only reads live AIS. AIS is dormant (key revoked weeks ago). It does NOT fall back to the web-scrape vessel count, so the headline transit metric renders as a broken-looking "--".

sigVessels: '--' sigVesselsSrc: 'FEED PENDING' sigVesselsDir: '▼' snapshot.scraped_vessel_total: 141 (data EXISTS, tile ignores it) ais_health.reason: silent_no_messages
Fix: point #sigVessels at scraped_vessel_total when AIS is dormant (the rpanel vessel-movement card already does this via applyAisDegradedMode — the signal-bar tile was never wired to the same fallback). Show "141 · WEB FEED" instead of "-- · FEED PENDING".
P1 · SHIP-BLOCKERDark-vessel tile shows "200" — GFW API cap, not a real count

renderGFW() sets the tile to encounters.length + loitering.length. The GFW query covers the WHOLE Persian Gulf (52-59.5°E, 24-28.5°N) over 30 days and hits the API's limit:200 cap. So "200" = "≥200 encounter events Gulf-wide, mostly legitimate fishing," displayed as "200 dark vessels near Hormuz."

sigDark: '200' gfwState: { enc: 200, lor: 0, loaded: true } ← exactly the API limit GFW geometry: entire Persian Gulf · 30-day window
Fix: (a) tighten the GFW geometry to a Hormuz-proximate bbox (~55.5-57.5°E, 26-27°N), (b) cap-aware: if result==limit, label "200+ (Gulf-wide, capped)" not a precise count, (c) ideally filter to vessels actually near the strait. Same lesson as dark_vessels=947: never show a number that reads as one thing but means another.
P1GFW feed half-broken — loitering 404, encounters times out

The loitering dataset returns 404. A direct POST to /api/gfw for encounters returns 524 (Cloudflare origin timeout). Only the cached/partial encounters path returns data, and that's the capped 200.

POST /api/gfw (encounters) → HTTP 524 (origin timeout) GFW [loitering-events-carriers:latest] → HTTP 404 Not Found console.warning logged on every page load
Fix: (a) the loitering dataset slug likely changed in GFW's API — verify current dataset names, (b) the 524 suggests the GFW upstream is slow; add a timeout + cached-fallback in functions/api/gfw.js so a slow GFW never blocks the tile, (c) if GFW is unreliable, consider degrading the dark-vessel tile to "unavailable" rather than showing a misleading number.
P2Incidents tile shows "—" (no live source)

#sigIncidents renders "—". incidents_30d is a hardcoded static field (58) declared in static_fields[], and the signal-bar tile doesn't render the static value. So the tile is permanently blank.

Fix: either wire a real incidents source (ACLED) or hide the tile entirely. A blank "—" tile in the signal bar reads as broken. Decide: real data or remove.
P2EIA official spot $10 below live futures — large gap, correctly dated but worth a glance

Live Brent (futures, Stooq/Yahoo) = $92. EIA official daily spot = $102.75 "as of 2026-05-26". Oil fell ~$10 in two days. The dashboard labels the official as dated, which is honest — but a $10 gap between the headline live price and the "official" reference can confuse a viewer.

live Brent: $91.7 (Yahoo $91.75 ✓) brentOfficial: $102.75 asOf 2026-05-26 (EIA daily, correct but 2 days lagged)
Fix: ensure the dashboard visually subordinates the official reference (smaller, clearly "EIA daily · 2 days ago") so the $92 live price is unambiguously the headline. Currently labeled correctly; just verify hierarchy in the redesign.
